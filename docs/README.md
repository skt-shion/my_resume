### トランザクションメール配信管理システム刷新（ニフティ株式会社: 2025/9 ~ 2026/2）

【担当領域 / 役割】

本刷新プロジェクトの調整・設計・開発を牽引。

- **アーキテクチャ設計:** 既存プロバイダー（Marigold Engage+）上の複雑なメール配信アーキテクチャを抜本的に再設計。データ基盤（マスタ）から独立した配信基盤を構築（ベンダー伴走支援契約の調整を含む）。
- **バックエンド・ワークフロー開発:** FastAPIを用いた設定自動化REST API、およびPrefect 3.xを活用した分散データ処理ワークフローの開発。
- **インフラ・環境構築:** Terraformを用いたAWSリソースのIaC化、VPCネットワーク設計、K8s APIリソースのデプロイ。

【課題】

- 既存システムはデータ基盤の更新処理に依存しており、マスタエラーが意図しない誤配信を引き起こすリスクや多大なリカバリ工数が発生していた。
- メール配信設定作業が複雑で属人化しており、データ基盤チームへの依頼集中による運用ボトルネックが生じていた。

【工夫した点】

- **ベンダー協業と推進:** 配信プロバイダー提供元と伴走支援契約を結び、タスク分担と連携体制を構築。複雑化したアーキテクチャの抜本的改善を円滑に推進した。
- **セルフサーブ化:** メール配信設定、テンプレート管理、テスト送信などの既存業務をユーザー自身で完結できるフローを設計。依頼対応から脱却し、利用者が自律して動けるセルフサービス型の配信管理システムを提供した。
- **モダンな高信頼アーキテクチャ（DDD・クリーンアーキテクチャ・非同期処理・分散アプローチ）:** Pythonの非同期I/O（async/await）とPrefectの分散型オーケストレーションを駆使し、高トラフィック、大量データの高スループット処理を実現。また、MongoDBにドメイン駆動設計（DDD）の思想を取り入れた3次元モデルとクリーンアーキテクチャを採用し、柔軟かつ網羅的なデータ管理と変更容易性を両立した。
- **可観測性と耐障害性:** Prefectのイベント駆動システムを活用したリアルタイムな障害検知・自動リカバリの仕組みを構築。加えて配信プロバイダー（Marigold Engage+）のユニークキー制約や配信ログのS3エクスポートを併用し、クリティカルな誤配信の防止と配信結果の容易な追跡を実現した。
- **セキュリティ対策:** 内部通信を含むすべての通信をTLSで保護するように構成し、データ連携に使用する外部ストレージはAmazon S3を選択。サーバーサイドの暗号化と最小権限のIAMポリシーでメールアドレスや顧客属性データを保護するなどでセキュリティ要件を担保した。
- **品質保証と開発環境整備:** Github Actionsで単体テストと統合テストのCI化による品質保証、Pydantic/mypyによる型安全の徹底、DevContainer/uvによる開発環境標準化を実施。

【成果】

- **障害波及リスクの排除:** マスタデータ連携から切り離した独立アーキテクチャにより、データ基盤障害によるメール誤配信リスクを根本から解消。
- **運用工数の劇的削減:** 複雑なメール配信設定作業をAPIでインターフェースとして公開することによりセルフサーブ化を実現し、属人性を排除。月間のトータル運用・障害対応工数を約80%削減（約5人日/月 → 約1人日/月）、設定業務等の運用工数を「2人日 → 0.2人日」へ大幅に削減することに成功。
- **プロバイダー変更を容易に:** ドメインモデル、アプリケーション設計で配信プロバイダーを抽象化することで、将来の切り替えや複数プロバイダー併用などの変更に柔軟に対応可能な作りにした。

【担当領域 / 役割】

4名体制で日常運用を継続しつつ、「データカタログ整備」「BI基盤移行」「データ基盤連携自動化」を牽引。

- **運用・保守:** 問い合わせ対応、EKS・Prometheus等のシステム基盤運用、dbtアップグレード対応。
- **BI基盤移行:** Tableau ServerからTableau Cloudへの資産移行・棚卸し。
- **データカタログ整備:** データカタログのプレ公開と、ヒアリングを通じたメタデータ拡充。
- **データ基盤連携自動化:** GitHub ActionsとPythonスクリプトによるCI/CD自動化ワークフローの構築。


【課題】

- データ連携案件で仕様変更が発生すると手戻りしやすく、工数負荷が増大の一要因となっていた。
- Tableau Serverの運用負荷およびクラウドコストの増大。
- カタログのマニュアル運用停止によるデータの信頼性・検索性低下と、問い合わせ工数の増大。

【工夫した点】

- **セルフサーブ化:** データ連携依頼を自動化し、Githubイシューベースでユーザー自身で完結できる仕組みの設計。

- **データ分析協業**: Tableau Cloud移行に伴い、多数のTableauワークブックを管理するデータ分析部隊と密に連携し、移行前のワークブック棚卸しや移行後の動作確認を協力して実施した。関係強化と安全な移行・クローズを両立。
- **現場に効果を:** カタログ公開では効果を一早く現場に届けるために、まずはプレ公開の形式で身近なデータ分析部隊向けに公開し、フィードバックを繰り返して改善を繰り返すスモールスタート方式を採用した。

【成果】

- **Tableau Cloud移行:** 既存資産の最適化と旧環境の安全なクローズを約3カ月間で完遂。
- **セルフサーブ型基盤の確立:** データ連携自動化により、データ基盤チームの作業工数を大幅に削減。
- **データカタログのプレ公開:** データ分析部隊を中心にプレ公開を実施し、フィードバックを得ながらメタデータ拡充を継続。データの信頼性・検索性向上と問い合わせ工数の削減に寄与。

### データ基盤運用 2024年通期（ニフティ株式会社: 2024/04 ~ 2025/03）

2023年下期に実施した「運用の見える化／監視・アラート整備／ドキュメント整備」を継続しつつ、通期を通じて発生した問い合わせ・定期作業・障害対応の処理と、運用改善を推進。

【担当領域 / 役割】

- **運用・保守（問い合わせ／トラブル対応／システム）**
	- データ基盤に関する社内問い合わせ対応（例: 新規データ連携、既存データスキーマ更新、その他データに関わる雑多なもの）
	- データ基盤関連システム・ツール（Marigold Engage+など）の問い合わせ対応
	- 定期作業（月次データ更新やアカウント棚卸など）
	- データパイプライン、システムアラート対応
	- モニタリング・オブザーバビリティ（Prometheusスタック）基盤運用
- **データパイプライン運用改善（障害復旧の迅速化・再発防止）**
	- ETL/ELTパイプラインワークフローバグ修正、パフォーマンス改善、リファクタリング
		- 日次・定期ワークフローの障害要因の切り分け、復旧、恒久対応の検討
		- 運用時に必要な情報（手順、依存関係、担当、実行方法）の整備・更新
- **ドキュメント整備**
	- 新人向け開発ガイドライン作成（開発環境構築、dbt Coreチュートリアル、パイプライン開発フローなど）
	- ワークフロー開発ガイド、リカバリ手順を全リポジトリに記載
	- dbt Core, EKSクラスターのアップグレードガイド、手順を整備し、実施履歴の記録と一元管理

【課題】

- 障害発生時に、原因調査・影響範囲特定・復旧判断に必要な情報が散在しており、復旧対応が長期化しやすい状態。
- データ利活用の現状が不透明
	- データはあるが果たしてそれが活用され、実益に結びついているのかが不透明な状態
	- SQLが書ける人に可視化・分析タスクが集中する傾向ありなど
- 組織的な構造に伴うデータ分析部隊との分断化
- リスクと非効率性を伴うデータパイプライン開発
	- 開発環境、テスト環境が存在せず、本番環境でテストをするかぶっつけ本番で祈るしかない
	- テストコードも当然存在しない
	- 結果的に構文誤りや設定ミスによる無駄なPRが多く発生し、PR本来の意味を為していない状態
	- デプロイすることが常に危険を伴う状態
- データパイプライン、データフローの不透明性
	- どこからデータが生じ、どういうプロセスを経た結果、どこにどんなデータが配置されるのかを把握する手段がソースを見るか、逐一ドキュメント化するしかない状況
	- 結果として障害対応、問い合わせ対応の長期化に繋がってしまっている
- ワークフロー同時実行制限と実行環境のリソースひっ迫によるボトルネック
	- コストパフォーマンスの観点から垂直スケーリングの限界に到達
	- クラスターの総物理コア数によってワークフローの同時実行数が制限
	- 特に多数のワークフローがドミノ式に失敗する大規模な障害対応時には上記がボトルネックとなり、障害対応の長期化に拍車をかけている状態
- 問い合わせ・定期作業・突発対応の比率が高く、改善活動にまとまった時間を確保しづらい。

【工夫した点】

- 出勤打刻後、デイリースクラムでダッシュボード、アラートチャンネル、コンソールを見る習慣を付け、現在のシステムの状態を把握する癖を付けた。異常の予兆把握、障害対応への耐性から生じる復旧時間短縮など運用面において様々な副次効果を得ることができた。
- デイリースクラムで直近発生した問い合わせを共有し、反復して発生する作業があればドキュメント化するタスクをバックログに登録することでチームの共有知を資産化。
- ドキュメント整備は「運用で実際に困った箇所」から優先して更新。「使える情報」（実行方法、依存関係、データフローなど）が載ったドキュメントだけを残し、誰も見ない不要なドキュメントはアーカイブする運用に寄せることで資産の負債化を防止する体制を構築した。
- オーケストレーションエンジン移行について、現状抱えている課題を本当に解決できるのかをイメージではなく、実際に検証。肌感覚で実感することでチーム全体の納得感を醸成し、現実の成果に結びつけられるようにした。

- **構築・検証:** AWS環境構築、各種サービス連携設定、サーバーアプリケーションの起動・内部動作確認。
- **進捗管理:** 全体的なタスク・スケジュール管理の遂行。

【課題】

- オンプレミス特有の複雑な依存関係や構成を正確に把握し、AWS環境で再現・最適化する必要があった点。
- 海外デベロッパー、社内開発チーム、運用チームなど多岐にわたるステークホルダーのタスクと工数を見積もり、サービス停止を最小限に抑えた移行スケジュールを完遂しなければならなかった点。

【工夫した点】
- 移行初期段階でオンプレミス環境のシステム・ネットワーク全体構成を詳細に把握することに注力し、移行における課題点の早期洗い出し。これにより設計フェーズが盤石になり、リスクを分散させる段階的な移行計画を策定することができた。
- 変更箇所は少ないほどリスクが下がる。移行リスクを最小限に抑えるため、既存の環境・構成を可能な限り同等レベルで再現するよう努めた。
- 同等の環境・構成にプラスアルファで、マネージドサービスで提供される機能を活用して可用性と運用負荷を緩和できるような設計を心がけた。
- 関係各所調整と移行リスクを踏まえたタスク・スケジュール管理を行うことで大きなトラブルもなく、ある程度の余裕をもって移行を完遂することができた。
- AWS移行後のランニングコスト増大を防ぐため、設計段階から消費リソースに見合う適切なインスタンスタイプ、ストレージを選定し、クラウドコスト最適化に努めた。
- 移行日直前に直属の上司によるレビューを実施してもらい、安全な移行に係る見落とし点やリスク要因の洗い出しが功を奏した。（DNSレコードのTTL問題）

【成果】

- 2タイトルのAWS移行を大きなトラブルなく完遂。
- 移行後の継続的な構成最適化により、オンプレミス時代と比較して50%以上のランニングコスト削減を実現した。
- 移行作業だけでなく、管理業務（タスク・スケジュール管理）を経験したことで、プロジェクト遂行能力を大きく向上させた。
