# （正式版・簡潔版）

## 1. これまでの経験の中で、ビジネスへインパクトを与えた経験について

最も大きなビジネスインパクトを出せた実績は、2025年後期〜2026年前期に主導したトランザクションメール配信システムの刷新です。旧システムは日次バッチで会員マスタを更新する方式に依存しており、更新が反映される前に配信すると配信漏れが起きるリスクを抱えていました。加えて配信設定の変更は都度個別開発が必要で、依頼がデータ基盤チームに集中し属人化していました。

本刷新では会員データをオンデマンドAPIで取得する方式に切り替えてマスタ依存を解消し、配信設定の変更をGitHub PRベースで自動化して利用者自身が完結できるセルフサーブ型に転換しました。対外的には配信障害の防止と復旧時間の削減、配信開始までのリードタイム短縮を実現し、対内的には月間の運用・障害対応工数を約80%削減（5人日/月→1人日/月）、設定業務の工数を2人日から0.2人日に圧縮しました。

ワークフロー基盤をPrefect+EKSへ移行した際も、複数の効果がありました。実行プロセスをEKS Fargateに置き換えて必要な時にリソースを調達する方式にしたことで、常時稼働させていた大型VMが不要になり、クラウドコストを月間40〜50%削減。分散実行環境への移行そのもので、クラスターの物理コア数に縛られていた同時実行数の制約も解消しました。また、本番環境しかなくテストのために本番デプロイが必須だった状況も解消し、Prefectの強みであるローカルでの開発・テスト・デプロイ検証が可能になったことで開発生産性も向上しました。

また、社内システムとのデータ連携も、Slackワークフローでの申請から検証・デプロイまでを自動化し、依頼者自身で完結できるセルフサービス型の運用フローにしました。この仕組みはdbtを中核に据え、申請内容をdbtのソース/プロパティ定義として管理し、CIでdbt compile/runを検証、マージ後はdbt run/test/docsまで自動実行します。狙いは工数削減そのものより、連携と同時にテーブル・カラムの説明や更新頻度などのメタデータが必ず付随する状態を作ったことです。これは過去にメタデータ整備が後回しになり負債化した経験を踏まえた設計で、ここで保証したメタデータは現在運用中のデータカタログ（OpenMetadata）にそのまま連携され、データ連携の入口から組織全体のデータガバナンスまで一貫した思想で取り組んでいます。

## 2. 技術を高めることに加えて、今後届けたい価値について

技術を高める先で見据えているのは、「データエンジニアを介さずにメタデータが成長し続け、非アナリストや生成AIエージェントも自ら必要なデータを見つけて使える」状態を作ることです。データエンジニアが仲介役として入り続ける限り、データ活用のスピードと範囲はチームの工数で頭打ちになります。この制約を解消し、人間だけでなく将来の生成AIエージェントも含めたデータ消費者が自律的に価値を生み出せるセルフサービス型のプラットフォームを作ることに、キャリアの軸を置きたいと考えています。

この価値観の出発点は、2023年下期から2024年にかけて、それまで分断されていたデータ分析部隊と約1年かけて信頼関係を築き、協業体制を作ったことにあります。合同スクラムイベントでの対話や全社アンケート・インタビューを通じて、現場のデータ消費者が実際に何に困っているかを直接聞き取りました。見えてきたのは、メタデータ不足、データの信頼性への不安、データ分析のハードルの高さ、データカタログへの需要でした。この現場の声が、以降の実績の出発点です。メール配信刷新では配信設定業務をAPI化してセルフサーブ化し、データ連携自動化では依頼者自身が連携を完結できる仕組みを作りながら、テーブル・カラムの説明や更新頻度などのメタデータが必ず付随する設計を組み込みました。これが現在推進しているデータカタログリニューアルに直接つながっており、データ連携の入口から組織全体のガバナンスに浸透させるという一貫した思想で取り組んでいます。

この状態を実現するために、自分に足りないと考えているのは3点です。1つ目は大量データを扱う分散処理の技術的バックグラウンドで、パイプライン開発における分散処理と、BI活用を見据えたクエリパフォーマンス重視のデータモデリング・データの分散配置設計を体系的に習得したいと考えています。2つ目はメタデータ管理そのものの設計・運用知見で、メタデータを「整備して終わり」にせず時系列で自律的に成長させる仕組みづくりです。3つ目はデータ品質・鮮度保証の仕組み化で、データ消費者がデータエンジニアに確認せずとも信頼できるかどうかを判断できる状態を作ることです。この3つは、メタデータが自律的に成長し続ける状態を支える土台だと捉えています。

生成AIの活用が進む中で、データの「消費者」は人間だけでなく、自然言語でデータを探索し意思決定を支援する生成AIエージェントにも広がっていくと考えています。現在推進しているデータカタログでは、生成AIエージェントがメタデータにアクセスするためのMCPインターフェースを標準搭載できる製品を選定しました。これは技術トレンドへの追従ではなく、「データエンジニアを介さずに済む状態」を人間だけでなく生成AIエージェントにも広げるための布石です。技術力を高めることそのものが目的ではなく、この価値を事業やデータ消費者に届けるための手段だと考えています。
