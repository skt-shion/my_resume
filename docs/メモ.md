# メモ：ビジネスインパクトと今後届けたい価値

1. データ活用やシステム改善を通じて、業務改善・売上向上・工数削減などビジネスへインパクトを与えた経験はあるか
2. 技術を高めることに加えて、今後データエンジニアとして事業やデータ消費者にどのような価値を届けていきたいか

参照元: [README.md](README.md)（職務経歴書）、[自己紹介.md](自己紹介.md)、[データカタログリニューアル.md](データカタログリニューアル.md)

## 1：ビジネスインパクトの回答構成

3事例を役割分担する構成で組み立てる。売上向上に直結する実績は現時点でないため、**工数削減・リスク低減・データ品質の構造的解決**を3本柱にする。

### メイン事例：トランザクションメール配信システム刷新
- 月間運用工数80%減（5人日/月 → 1人日/月）
- 設定業務 2人日 → 0.2人日
- 詳細は「メール配信刷新の追加背景」セクション参照

### サブ事例：ワークフロー基盤移行（Prefect + EKS移行）
- クラウドコスト40〜50%減
- 開発生産性向上（同時実行制約の解消）

### 橋渡し事例：データ連携自動化
- 工数は1件あたり2人日 → 0.5〜1人日（50〜75%減）。ただし月1〜2件と少量のため集計インパクトは小さい
- 本質的な価値は工数削減ではなく、**データ連携と同時にメタデータ（テーブル・カラム説明、担当チーム、連絡先、更新頻度）が100%付随する仕組みを作ったこと**
- README 2025年通期の反省点「メタデータ標準化の初動が遅れた」を、次のプロジェクト設計で解決した実例
- この構造的に保証したメタデータは、データカタログリニューアル（OpenMetadata採用、既に早期リリース済み、現在はドメイン担当者・データ基盤担当者へのアクセス誘導設定を投入中のフェーズ）にそのまま連携される設計になっており、「データ連携の入口から組織全体のデータガバナンス層まで一貫した設計思想」として語れる
- 質問2への橋渡しとしても使う

**方針**：メイン事例で数字を主役にし、サブ事例で補強、橋渡し事例は「工数だけでなく資産の質を保証した」という質的な話として短く添える。

## 2：今後届けたい価値の回答の核

### 価値の核
「データエンジニアを介することなくデータ資産に付随するメタデータが時系列を経て成長し続け、営業・企画チームなどの非アナリストや、データ分析を担う生成AIエージェントも、自身の意思決定に必要なデータを自ら見つけ出し、扱えるようになる」状態を作ること。

### 実績との接続
メール配信刷新（セルフサーブ化）→ データ連携自動化（メタデータの構造的付随）→ データカタログリニューアル（組織全体のガバナンス層への拡張）という一貫した流れとして語る。

### 技術軸（この価値を実現するために不足していると認識している3要素）
1. **ビッグデータ処理の技術的バックグラウンド** — Pythonによるデータパイプライン開発におけるpandas・Sparkを用いた大量データ処理の知見、BIツールからのクエリパフォーマンスを意識したデータモデリング手法と分散戦略（dist/sortkeyなど）
2. **メタデータ管理の設計・運用知見** — メタデータが自律的に成長し続ける仕組みづくり
3. **データ品質・鮮度保証の仕組み化** — データ消費者が都度確認せず信頼できる状態

### AIエージェントへの言及
データカタログでOpenMetadataを選定した理由の一つがMCP標準搭載（生成AIエージェント向けインターフェース）であることと絡めて、「人間だけでなく生成AIエージェントもデータ消費者として扱う」という将来観を明示する。

**方針**：「技術を高めること」と「価値を届けること」を分断せず、技術軸の3要素が価値の核を実現するための土台であるという因果関係で繋げる。

## メール配信刷新の追加背景（drawio由来）

### Before（旧アーキテクチャ）
- 会員DBから日次バッチETLで受信者DBを更新する方式 → マスタ更新が反映される前に配信すると配信漏れが発生するリスク構造
- 配信設定の変更は「個別開発・設定スコープ」として都度アプリ改修で対応 → 属人化
- 受信者リストはファイルベースでWorkflow Orchestration経由でロード

### After（新アーキテクチャ）
- 会員APIからオンデマンドでメールアドレスを取得（配信設定.json）し、マスタ依存を解消
- 配信設定の変更はGitHub PRベースで自動化（GAS → PR作成 → Github Actions WFでValidation & Auto-merge → Deploy）。属人化解消と変更容易性を両立
- Prefectワークフロー（Validation → Preprocessing → Transformation → Queuing → Dispatch → Recovery）とAmazon EventBridge + Step Functionsによるイベント駆動処理
- 失敗時のFailure Notification、送信履歴のExportなど可観測性・リカバリ機能を追加
- マルチAWSアカウント構成（UMPS / Central / データ基盤）をTransit Gateway経由のPeering・VPCエンドポイントで接続し、EKS上で配信設定管理APIを稼働。MongoDB、SQS/SNS、Secrets Managerを利用

### 価値分類
- **対外向け**：より安全なメール配信と可観測性の確保、メール配信障害（配信漏れ・誤配信）防止・復旧時間削減（どのIDで誰宛の配信に失敗したか専用チャンネルで通知）、配信開始までのリードタイム削減（任意タイミングでテスト配信・リリース可能に）
- **対内向け**：属人化解消、事業側の要求変化への対応力（配信可能メールアドレス種別追加、メール配信プロバイダー切り替えなど）

この対外/対内の2軸整理は、技術的な工夫を「誰にとっての価値か」に翻訳する際の骨格として使う。
